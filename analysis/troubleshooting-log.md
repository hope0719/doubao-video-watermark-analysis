# 豆包视频去水印排查总结

> **关键词：豆包去水印排查、API测试、URL参数操控、CDN etag验证、开源项目分析、get_play_info失效**

> 2026-07-04 | 完整记录所有尝试、测试数据和最终结论

---

## 一、背景

- 项目：豆包视频/图片去水印工具
- 平台：微信小程序 + Edge 浏览器插件
- 核心假设：`get_play_info` API 返回 `videoweb.doubao.com` 域名 = 无水印原片
- 问题：从某个时间点开始，所有下载的视频都带有水印

---

## 二、去水印核心原理

### 2.1 唯一有效的 API

```
POST https://www.doubao.com/samantha/media/get_play_info?aid=497858&device_platform=web&language=zh-CN
Body: { "key": "<video_id>" }
Headers: origin/referer=user-agent 伪装为浏览器
```

### 2.2 响应结构（期望）

```json
{
  "original_media_info": {
    "main_url": "https://v26-videoweb.doubao.com/.../xxx.mp4"  // ← 无水印
  },
  "media_info": [{
    "main_url": "https://v26-videoweb.doubao.com/.../xxx.mp4"  // ← 720p 兜底
  }],
  "poster_url": "..."
}
```

### 2.3 获取 video_id

| 场景 | video_id 来源 |
|------|-------------|
| video-sharing 页面 | URL 参数 `?video_id=v0269...` |
| chat 页面 | inject.js 拦截 `/im/chain/single` API 响应 `video.vid` |

## 三、已穷尽的所有测试

### 测试用视频
```
video_id: v0269cg10004d946i5iljhtf2dunr5e0
share_id: 49152711347982082
URL: https://www.doubao.com/video-sharing?source_type=mobile&share_id=49152711347982082&video_id=v0269cg10004d946i5iljhtf2dunr5e0
```

### 测试 1：调用 get_play_info API

```
curl -X POST 'https://www.doubao.com/samantha/media/get_play_info' \
  -d '{"key":"v0269cg10004d946i5iljhtf2dunr5e0"}'
```

**结果：**
- `original_media_info.main_url` → `v26-videoweb.doubao.com`，路径哈希 `7ef23d72233...`
- `media_info[0].main_url` → **完全一致**的 URL
- **两个字段不再有区分**，都带 `lr=video_gen_watermark_dyn`
- 下载后确认：**有水印**
- 文件大小：843,802 字节（cs=0 版本）

### 测试 2：调用 get_video_share_info API

```
curl -X POST 'https://www.doubao.com/creativity/share/get_video_share_info' \
  -d '{"share_id":"49152711347982082","vid":"v0269cg10004d946i5iljhtf2dunr5e0","creation_id":""}'
```

**结果响应结构：**

### 测试 3：URL 参数操控

| 测试 | 修改方式 | 结果 |
|------|---------|------|
| 去掉 `lr` 参数 | 删除 `lr=video_gen_watermark_dyn` | etag 不变，同一文件 ✅ |
| `lr=none` | 替换为 `lr=none` | etag 不变，同一文件 ✅ |
| 强制 `cr=7&dr=3&cs=4` | 替换为 share_info 的参数值 | etag 不变，同一文件 ✅ |
| 删除 `download=true` | 去掉该参数 | etag 不变，同一文件 ✅ |

**结论：CDN 文件由路径哈希决定，URL 参数只影响签名/缓存，不影响实际内容。**

## 六、根因分析

### 6.1 不是我们代码的错
- 微信小程序（完全相同的 API 流程）现在也带水印 → 问题在字节服务端
- 浏览器实际播放画面也带水印 → 不是 DOM/API 返回值的差异
- 所有 mp4 URL 的 etag 相近但不完全相同 → 是不同的编码版本（cs=0 vs cs=4），不是有/无水印的区分

### 6.2 最合理的解释

1. **CDN 只存一份文件**——视频编码时就已经叠加了水印
2. ByteDance 某个时期可能曾存在**无水印 Bug**（CDN 旧节点或旧编码器没有正确添加水印）
3. 现在 Bug 已修复 → 所有渠道都返回带水印版本
4. `lr=video_gen_watermark_dyn` 只是一个**标签参数**，CDN 已经忽略它
5. `cs` 参数区分**不同编码配置**，不是有/无水印的区分

### 6.3 验证 ft= 参数的真相（2026-07-04 新增）

**假设：** `ft=` 是边缘节点的权限签名 token，控制 CDN 叠加水印 vs 直接返回原片。

**测试：** 4 种 `ft=` 变体（完整值 / 去掉 / 空值 / 随机值）返回**完全相同**的文件。

| 测试 | etag | content-length |
|------|------|----------------|
| 原始 `ft=vTVAHK...` | `5bd9650c...` | 843,802 |
| 去掉 `ft=` | `5bd9650c...` | 843,802 |
| `ft=` 空值 | `5bd9650c...` | 843,802 |
| `ft=AAAA` | `5bd9650c...` | 843,802 |

**结论：** `ft=` 只是 CDN 缓存 nonce，**不是权限控制参数**。水印不在边缘叠加。之前的"边缘水印叠加"假设不成立。

### 6.4 穷举后的唯一结论

**所有 URL 参数均不影响水印有无。** 实际方案是：

> AI 生成无水印原片 → 编码时叠加水印至每一帧像素 → CDN 只存一份带水印文件 → 所有 URL 参数均为缓存签名 → 客户端无法绕过

这是在编码层物理叠加，不是分发层逻辑控制。

### 6.5 ByteDance 一定有无水印原片

**这是一个非常关键的逻辑推论：字节跳动一定能无水印原片，只是不公开。**

理由如下：

1. **AI 模型（Seedance 等）生成的原始视频就是无水印的**。水印是后处理叠加上去的，不是模型输出时自带的。
   
2. **创作者需要无水印版本**。用户用豆包生成的视频，如果要发到抖音/B站/小红书，带豆包水印的视频无法通过审核。平台必须提供"创作者无水印下载"能力——这是AI视频平台的基本功能。

3. **商业逻辑要求**。如果字节自己都无法提供无水印版本，那创作者根本无法使用自己的产出，整个AI视频生成业务就不成立。

4. **水印叠加的常见工业流程**是：原始文件 → 编码 → 叠加水印 → 分发。原始文件在叠加前一定存在于某个存储中。

### 6.4 无水印原片的可能位置

| 位置 | 说明 | 我们是否可访问 |
|------|------|---------------|
| **AI 生成后、水印叠加前的缓冲存储** | 生成流水线中的临时文件 | ❌ 不可达，秒级清理 |
| **对象存储（内部归档）** | 完整的原始视频存档，供创作者后台下载 | ❌ 需要创作者 OAuth Token |
| **创作者专属 CDN** | 登录后通过作者身份校验的临时下载 URL | ❌ 需要登录态 + 签名 |
| **内部分发通道** | 抖音创作者服务平台 / 巨量引擎的API | ❌ 需要商业授权 |

### 6.5 为什么所有公开 API 都拿不到？

因为我们之前调用的是**公开分发 API**（`get_play_info`、`get_video_share_info`），这些 API 专门设计为返回"分享用的带水印版本"。

无水印版本在**完全不同的 API 路径**上：
- 需要创作者身份（OAuth token）
- 需要更高的权限级别
- 走的是内部/半内部的服务链路，不是公开链路

**分水岭：公开分发 vs. 创作者下载。我们之前一直在"公开分发"这条路上打转。**

### 6.3 排除的猜测

| 猜测 | 为什么排除 |
|------|-----------|
| 需要登录/鉴权 | API 不需要 cookie也能正常返回，不需要登录 |
| URL 水印参数 | 去掉 lr、改 lr=none 均无效 |
| CDN 域名差异 | 所有域名（v9/v26/show）均返回相同文件 |
| cs=0 vs cs=4 | 文件不同但水印相同 |
| 浏览器播放 vs API | 下载后完全相同 |

## 七、经验总结

### 7.1 关于「去水印」
- 豆包视频水印是**服务器端编码时叠加**的，不是在客户端添加的
- 如果字节不返回无水印版本，任何客户端方案都无法去除
- 之前在（v7-v8）时间点能工作，可能是 CDN 配置错误遗留

### 7.2 关于 API 选择
- `get_play_info` 和 `get_video_share_info` 返回的是相同的带水印文件
- 两个 API 同时改为主流，不再是隐藏的`vod`域名

### 7.3 关于视频
- 即使 API 返回的 URL 参数中有 `download=true`，也并不能获得无水印版本
- 水印不在传输层，在编码层

---

## 八、剩下的选择

| 方案 | 可行性 | 备注 |
|------|--------|------|
| 放弃视频去水印 | ⭐ 容易 | 图片去水印仍可保留 |
| 客户端 AI 修复 | ❌ 不可行 | 需要服务端跑 FFmpeg + AI，延迟高，成本高 |
| 屏幕录制 | ⚠️ 勉强可行 | Playwright 打开页面→自动播放→录制→保存。产出"录制版"而非"原片"。失去了"原片"的意义 |
| 等 Byte 变动 | ⚠️ 无期 | 不可控 |

## 九、图片去水印仍然有效

需要强调：**图片去水印是可以工作的**！

图集页面（`/thread/`）返回的 `rc_gen_image/{32位md5}` URL 确实是**无水印原图**。这部分逻辑没有问题，可以继续保留。
