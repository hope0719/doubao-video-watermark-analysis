## 一、当前态势总结

### 核心困境

经过大量测试，**所有通过标准 API 获取的视频 CDN URL 下载后，视频画面内都含有水印**。无论是哪种调用方式、哪个 CDN 域名、什么 `lr` 参数，最终文件的水印都被编码嵌入画面中。

### 测试矩阵

| # | 测试来源 | CDN 域名 | lr 参数 | 文件 MD5 | 大小 | 水印 |
|---|---------|----------|---------|---------|------|------|
| 1 | 小程序抓包（3次） | v9-videoweb | unwatermarked | 不同视频不同 | ~598KB~741KB | ❌ 有 |
| 2 | H5 浏览器调 API | v9-videoweb | video_gen_watermark_dyn | d903cd06... | 819KB | ❌ 有 |
| 3 | H5 播放器页面 video source | v26-videoweb | video_gen_watermark_dyn | f72c11e7... | 653KB | ❌ 有 |
| 4 | 微信 UA + 完整 query 参数调 API | v26-videoweb | video_gen_watermark_dyn | d903cd06... | 819KB | ❌ 有 |
| 5 | doubao-nomark 开源项目解析下载 | v9/v26 | video_gen_watermark_dyn | d903cd06... | 819KB | ❌ 有 |

**关键结论**：上述 5 种方式下载的文件，虽然路径、域名、编码各不相同，但**所有文件都有可见水印**。

## 二、已证实的发现

### 2.1 `lr` 参数在 CDN 层面完全无效

- `lr=unwatermarked` 和 `lr=video_gen_watermark_dyn` 在相同路径下返回**完全相同的文件**（MD5 一致）
- 根本原因：CDN 缓存 key（`X-Kfc-Cachekey` 响应头）**只包含路径，不含查询参数**
- `lr` 参数仅用于服务端数据分析/计费，不影响文件内容

### 2.2 不同 CDN 域名对应不同编码版本

| 域名 | 编码 | cs 参数 | 典型码率 |
|------|------|---------|---------|
| v9-videoweb | H.264 | cs=0 | ~1257kbps |
| v26-videoweb | H.265 | cs=4 | ~1002kbps |
| 小程序专用路径 | H.265 | cs=4 | ~918kbps |

- v9 提供 H.264 版本（兼容性好）
- v26 提供 H.265 版本（压缩率高，文件更小）
- 小程序请求的路径编码参数（`cs=4 ds=4`）与 H5（`cs=4 ds=3`）略有不同
- **所有版本都含有水印**

### 2.3 `feature_id` 标识不同场景

| 场景 | feature_id |
|------|-----------|
| 小程序下载/播放 | `069767e0b4f5d9d87fcf68b96cd224a7` |
| H5 播放器 | `069767e0b4f5d9d87fcf68b96cd224a7`（同） |
| H5 API 返回 | `e38567d78da7ae34faf3833d9e13c66f` |

`069767e0...` 是"播放场景"标识，`e38567d7...` 是"API 下载场景"标识，但两者都有水印。

### 2.4 不同请求方式返回不同域名

| 调用方式 | API 返回的 URL 域名 |
|---------|------------------|
| 普通浏览器 UA + 无 query 参数 | v9-videoweb |
| 微信 UA + 完整 query 参数 | v26-videoweb |

**用微信 UA 调用 API 可以拿到 v26 域名的 URL**，但文件内容仍然相同（MD5 一致）。

## 三、doubao-nomark 开源项目分析

### 项目信息

| 项目 | 说明 |
|------|------|
| 仓库 | https://github.com/ihmily/doubao-nomark |
| 核心能力 | 从豆包对话链接提取无水印图片和视频 |
| 技术原理 | 逆向抓取豆包官方 API 返回的无水印资源 |
| 部署方式 | Python/FastAPI API 服务、Docker、Edge 扩展、油猴脚本 |

### 核心代码分析

视频解析的核心调用：

```python
# doubao_parser/video.py
response = await client.post(
    "https://www.doubao.com/samantha/media/get_play_info",
    params=params,  # 13 个 query 参数
    headers={       # 微信 PC 版 UA
        "User-Agent": "Mozilla/5.0 ... MicroMessenger/7.0.20.1781 ... Flue",
        "origin": "https://www.doubao.com",
    },
    json={"key": vid},  # video_id
)
result["data"]["original_media_info"]["main_url"]  # ← 这就是最终的 URL
```

### 实测验证

我们完整运行了 doubao-nomark 的 `doubao_video_parse()` 函数：
- ✅ API 调用成功，返回正常
- ✅ 获取到 `original_media_info.main_url`
- ✅ 下载视频成功，**MD5 和我们之前 API 调用完全一致**
- ❌ **视频画面中有水印**

### 结论

**doubao-nomark 项目无法去除视频水印。** 它使用的 API 和参数与我们之前的方式本质上相同，返回的视频文件也完全相同（均有水印）。该项目的主要价值在于**图片去水印**，视频功能可能仅在特定场景下有效或尚未修复。

## 四、第三方小程序「云龙清图助手」分析

### 基本信息

| 信息 | 内容 |
|------|------|
| 名称 | 云龙清图助手 |
| 类型 | 微信第三方小程序 |
| 使用方式 | 输入豆包视频链接 → 解析 → 保存到相册 |
| 用户验证 | 保存到相册前已无水印 |

### 抓包结果

通过 Stream 抓包工具在小程序运行时的网络请求：
- **只抓到视频下载请求**（v9-videoweb 域名，`lr=unwatermarked`）
- 视频下载请求**前只有微信广告请求**，无 API 调用
- 搜索 `unwatermarked` 关键词也未发现其他请求

### 解释模型

既然抓包未发现 API 调用，小程序如何获取视频 URL 的可能性：

| 可能性 | 说明 | 可信度 |
|--------|------|--------|
| **A. 客户端裁剪** | 小程序播放器裁剪了水印边缘区域，保存时也做裁剪 | ⭐⭐⭐⭐ |
| **B. Protobuf API** | API 走 protobuf 协议，Stream 显示为二进制被忽略 | ⭐⭐⭐ |
| **C. 客户端 AI 去水印** | 小程序内置 AI 模型本地去水印 | ⭐⭐ |
| **D. 第三方服务器代理** | 第三方服务器代为调用内部 API 并返回处理后的视频 | ⭐⭐⭐ |
| **E. WebSocket 推送** | 通过 WebSocket 推送 URL，未在 HTTP 请求中体现 | ⭐⭐ |

### 最可能的解释

**方案 A（客户端裁剪）是目前最合理的解释。**

理由：
1. 抖音/豆包系平台的水印通常位于画面边缘（右上角或右下角）
2. 播放器可以用 `object-fit: cover` 或 Canvas 裁剪掉边缘
3. 保存到相册时也可以做同样的裁剪
4. 小程序抓包只有 v9-videoweb 的视频下载请求，没有额外 API
5. 第三方小程序不需要调专用的内部 API

## 五、水印去除的可行方向

### 方向一：画面裁剪（短期可行）

**原理**：豆包水印通常位于画面固定位置（右上角/右下角），可以通过裁剪边缘区域去除。

**实现方式**：

```bash
# 通用裁剪命令（参数需根据实际水印位置调整）
ffmpeg -i input.mp4 -vf "crop=1240:720:40:0" -c:a copy output.mp4
```

**优点**：实现简单，无需 API 改造
**缺点**：损失画面边缘像素，如果水印位置变化则需要动态检测

**在我们的 Edge 插件 / 小程序中的实现方案**：
1. 下载原始视频到本地
2. 使用 Canvas 或 WebCodecs API 在浏览器端裁剪
3. 保存裁剪后的视频

### 方向二：AI 去水印（长期可行）

**原理**：使用 AI 模型（如 OpenCV inpainting、深度学习模型）自动检测并修复水印区域。

**优点**：不损失画面，水印位置自适应
**缺点**：需要 AI 模型推理，计算量大

### 方向三：寻找真正的无水印 API（理论可能）

**现状**：
- 标准 API `samantha/media/get_play_info` 返回的都是有水印文件
- doubao-nomark 开源项目也无法绕过
- 小程序的 API 调用未抓到

**可能的突破口**：
1. 小程序中使用的 protobuf 协议 API
2. 小程序传递的额外参数（如 `net=5` 代表的特定场景）
3. 小程序中使用的特殊 Cookie/token

## 六、结论与建议

### 当前结论

1. **标准 API 路径不可行** — `/samantha/media/get_play_info` 返回的所有 URL 都有水印
2. **开源项目不可用** — doubao-nomark 无法去视频水印
3. **lr 参数无效** — CDN 不区分
4. **小程序机制未破解** — 云龙清图助手的 API 调用未抓到

### 推荐下一步

**短期方案（1-2 天）**：
- 实现画面裁剪功能（定位具体水印区域，做精确裁剪）
- 在 Edge 扩展中添加裁剪选项
- 在小程序中实现裁剪后保存

**中期方案（1-2 周）**：
- 用 Stream 重抓小程序，搜索 protobuf 请求
- 导出小程序的视频文件做二进制对比
- 尝试分析小程序的 React 组件或 JS 代码

**长期方案（视需求）**：
- 集成 AI 去水印模型
- 寻找第三方服务 API 接口
- 关注 doubao-nomark 项目更新
