## 二、明确遗漏的测试

### 🔴 遗漏 1：用真实登录 Cookie 测试登录态端点

**现状**：`doubao_login_probe.py` 和 `doubao_full_probe.py` 都已写好，但**从未用真实 Cookie 执行过**。

**用户 Monitor JSON 中有宝贵信息**：
- 用户的浏览器确实调用了 `/samantha/video/get_play_info` 并成功返回 200
- 但返回的 `play_infos[0].main` URL 仍然带 `lr=video_gen_watermark_dyn`
- **这意味着即使用户登录态，豆包自己的 API 也返回带水印视频** —— 这是一个非常重要的结论，但我们还没有通过主动调用验证

**需要做的**：
1. 从用户浏览器提取真实 Cookie（用户之前尝试过，但还没有成功导出可用的 Cookie 文件）
2. 用 Cookie 调用 `/samantha/video/get_play_info`，确认响应中是否有无水印 URL
3. 同时测试 `/samantha/video/query_video_gen_info`、`/alice/resource/watermark_task` 等端点

### 🔴 遗漏 2：CDN URL 直接修改 `lr` 参数

**现状**：我们测试了在 API 调用时修改 `lr` 参数，但**没有测试直接修改 CDN 视频 URL 中的 `lr` 参数**。

具体来说：
- API 返回的视频 URL：`https://v26-videoweb.doubao.com/...lr=video_gen_watermark_dyn...`
- 我们没试过直接请求：`https://v26-videoweb.doubao.com/...lr=no_watermark...`

**理论依据**：虽然之前的报告说"CDN 缓存 key 不含查询参数"，但这是针对 API 响应的 etag。对于视频文件请求本身，某些 CDN 配置可能会根据 `lr` 参数路由到不同存储路径。需要实际测试。

### 🔴 遗漏 3：`v6-show.douyinvod.com` 域名

**现状**：用户 Monitor JSON 中的视频 URL 使用的是 `v6-show.douyinvod.com` 域名，这是我们之前没有重点测试的域名。

### 🟡 遗漏 5：视频生成过程的实时 API 监控

**现状**：我们测试的都是"视频已生成完毕"后的 API 调用。但豆包在"视频生成中"可能有不同的 API。

**猜测**：视频生成完成后，会有一个"通知"或"状态更新"API 调用，此时返回的视频 URL 可能是临时性的（不同 `lr` 参数）。

**需要做的**：在点击"生成视频"到"视频生成完毕"的整个过程中抓包，而不仅仅是在分享页面抓包。

### 🟡 遗漏 6：手机 App API 逆向

**现状**：所有测试都是针对 Web 端。手机 App（iOS/Android）可能使用不同的 API 端点，这些端点可能有无水印版本。

**依据**：`FIND_MINIPROGRAM_API.md` 中提到，手机端抓到了 `lr=unwatermarked` 的 URL（虽然下载后仍然带水印，但说明手机端的参数传递方式不同）。

### 🟡 遗漏 7：确认水印是烘焙还是叠加

**现状**：我们一直假设水印是在服务端编码时烘焙到视频中的，但**没有实际下载视频文件并逐帧分析**。

**需要做的**：
1. 下载一个带水印的视频文件
2. 用 FFmpeg 提取第一帧和最后一帧
3. 检查水印位置是否固定（烘焙的水印位置固定，叠加层可能位置不同）
4. 检查是否可以在不重新编码的情况下移除水印（如果是独立轨道的话）

### 🟡 遗漏 8：特殊请求头测试

**现状**：所有测试都使用标准请求头。没有测试以下特殊头：
- `X-Requested-With`

- `sec-fetch-*` 头

## 三、剩余可行方向

### 方向 A：画面裁剪（最短期内可落地）⭐ 推荐

- **原理**：豆包水印一般固定在画面边缘（通常是右下角），通过裁剪可去除
- **实现**：`ffmpeg -i input.mp4 -vf "crop=1240:720:0:0" output.mp4`
- **优点**：不依赖服务端，实现简单，完全合法
- **缺点**：损失边缘像素，需预先知道水印位置
- **下一步**：下载几个样本视频，确认水印精确位置，计算裁剪参数

### 方向 B：AI 后处理去水印（效果最好，成本最高）

- **原理**：逐帧检测水印区域并用 AI inpainting 修复
- **工具**：Morph Studio（在线）、ComfyUI + ProPainter（自托管）
- **优点**：可处理动态水印、中央水印，效果最好
- **缺点**：需要 GPU / 付费额度，处理慢
- **下一步**：评估 Morph Studio 免费额度，或搭建 ComfyUI 工作流

### 方向 C：用真实登录 Cookie 完成登录态端点测试 ⭐ 高优先级

- **原理**：我们写好脚本但从未用真实 Cookie 执行。如果用户能提供有效 Cookie，可能发现不同结果
- **依据**：用户的 Monitor JSON 显示登录态下调用 `/samantha/video/get_play_info` 返回 200，但 `lr` 仍然是 `video_gen_watermark_dyn`。需要完整分析响应中是否隐藏着无水印 URL
- **下一步**：帮助用户从浏览器导出 Cookie，执行 `doubao_login_probe.py`

### 方向 D：持续监控 API 变化

- **原理**：豆包可能在未来版本中恢复 `original_media_info` 与 `media_info` 的区分，或者新增去水印 API
- **实现**：每日自动调用 `get_play_info`，比较响应格式变化
- **下一步**：写一个简单的监控脚本，部署到可定时运行的环境

### 方向 E：官方会员/创作者渠道（最合法）⭐ 推荐

- **原理**：豆包面向创作者或会员可能提供无水印导出能力
- **优点**：无损、合法、稳定
- **缺点**：可能付费
- **下一步**：在豆包 APP/网页中查看"导出/下载"高级选项，联系豆包客服

### 方向 F：研究第三方服务的服务端能力（调研级）

- **原理**：某 Android 下载工具在服务端完成了无水印解析，说明存在服务端能力
- **风险**：需要上传视频到第三方服务器，有隐私泄露风险；可能涉及版权/ToS 问题
- **下一步**：仅作为调研方向，收集公开文档，不直接集成
