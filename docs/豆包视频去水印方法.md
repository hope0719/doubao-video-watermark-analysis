# 豆包视频去水印方法

> 验证日期：2026-05-15
> 适用：豆包 Web 分享链接（`https://www.doubao.com/video-sharing?...`）
> 状态：✅ 可用

---

## 1. 背景

豆包用户分享出来的视频链接形如：

```
https://www.doubao.com/video-sharing?source_type=mobile&share_id=44993598315531266&video_id=v0269cg10004d82qodiljht41c5h83qg
```

直接打开能看到一个右下角带"豆包"水印的视频。早期常见的"去水印"做法是调
`/creativity/share/get_video_share_info` 拿 `play_info.main` 然后把 URL 中
`lr` / `watermark` 等参数删掉——**这套做法目前已失效**：

- 该接口现在统一返回 `lr=video_gen_watermark_dyn` 的 CDN 资源
- 水印是**预烧录到视频像素**里的（不是 CDN 转码时叠的）
- 删掉/替换 `lr` 参数后，CDN 仍然返回同一份带水印 mp4，字节数完全相同

## 2. 真正的无水印接口

通过反编译 `video-sharing` 页面的 webpack chunk（`46107.d0d72789.js`），
找到了 doubao 内部用于多媒体播放的统一接口：

```
POST https://www.doubao.com/samantha/media/get_play_info?aid=497858&device_platform=web&language=zh-CN
```

### 关键点

| 项 | 值 |
|---|---|
| 必需 query | `aid=497858`（不带会被服务端拦截，返回 `200 + content-type: text/plain + 空 body`）|
| 必需 body | `{"key": "<video_id>"}`（注意字段名是 `key`，不是 `vid`） |
| 是否需要登录 | ❌ 不需要 |
| 是否需要 `share_id` | ❌ 不需要（share_id 是另一套体系，反而会让接口报 `param invalid`）|
| Content-Type | `application/json` |

### 请求示例（Node）

```js
const https = require('https');
const body = JSON.stringify({ key: 'v0269cg10004d82qodiljht41c5h83qg' });

const req = https.request({
  hostname: 'www.doubao.com',
  path: '/samantha/media/get_play_info?aid=497858&device_platform=web&language=zh-CN',
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Content-Length': Buffer.byteLength(body),
  },
}, (res) => {
  const chunks = [];
  res.on('data', c => chunks.push(c));
  res.on('end', () => console.log(Buffer.concat(chunks).toString('utf8')));
});
req.write(body);
req.end();
```

### 响应结构

```json
{
  "code": 0,
  "msg": "",
  "data": {
    "media_type": "video",
    "media_info": [
      {
        "meta": {
          "height": "960",
          "width": "720",
          "format": "mp4",
          "duration": 10.08,
          "codec_type": "h264",
          "definition": "720p"
        },
        "main_url": "https://v26-videoweb.doubao.com/.../oYYA4Kkb0slh.../?...&download=true",
        "backup_url": ""
      }
    ],
    "original_media_info": {
      "meta": {
        "height": "1112",
        "width": "834",
        "format": "mp4",
        "duration": 10.05,
        "codec_type": "h264",
        "definition": "1080p"
      },
      "main_url": "https://v26-videoweb.doubao.com/.../e9fb6cb0e1f84cb99a01237a1007ca60/?...&download=true",
      "backup_url": ""
    },
    "poster_url": "https://...image",
    "playable_status": 1
  }
}
```

| 字段 | 说明 |
|---|---|
| `media_info[0].main_url` | **720p 无水印**预览版（H.264，约 1.5 Mbps） |
| `original_media_info.main_url` | **1080p 无水印源片**（H.264，5–6 Mbps，对应 App 内"高清下载"档） |
| `poster_url` | 封面图（带签名，1 小时左右过期） |
| `playable_status` | 1 表示可播放 |

URL 末尾的 `download=true` 是 doubao 直接提示这是为下载而生成的源片版本。

## 3. 水印版 vs 无水印版的对比

通过 `ffprobe` 实测同一 video_id 三种来源的差异：

| 来源 | CDN 路径 hash | 编码 | 分辨率 | 码率 | 大小 |
|---|---|---|---|---|---|
| `get_video_share_info.play_info.main` | `oQDCr4YI1BetAWbK0QkFwh1Erg4UAI3ysEsAQy/` | HEVC | 720x960 | 1398 kbps | 1.76 MB |
| `media/get_play_info` 720p | `oYYA4Kkb0slhEErDCQ1QeAwFIK4tIyU1IANBsU/` | H.264 | 720x960 | 1484 kbps | 1.87 MB |
| `media/get_play_info` original | `e9fb6cb0e1f84cb99a01237a1007ca60/` | H.264 | **834x1112** | **6103 kbps** | **7.67 MB** |

三个文件 CDN 路径完全不同，**说明水印是预烧录在独立文件里的**，单纯改 URL 参数无法去除。

## 4. 完整解析流程

要想同时拿到无水印 URL + 视频元信息（prompt、作者昵称、封面），需要并行调两个接口：

| 接口 | 用途 | 必需参数 |
|---|---|---|
| `POST /samantha/media/get_play_info` | 真正的视频 URL（无水印） | `{key: vid}` |
| `POST /creativity/share/get_video_share_info` | 拿 prompt + 作者 nickname | `{share_id, vid, creation_id:""}` |

工程上的建议：

1. 主流程以 `media/get_play_info` 为准，失败即报错
2. `get_video_share_info` 只用来补元数据，失败时静默回退到空字符串
3. 两个接口并行发出，给元信息接口一个短超时（≤1.5s）防止慢接口拖累整体响应

参考实现：[`utils/api.js`](../utils/api.js) 中的 `doubaoFetchPlayInfo` / `doubaoFetchShareMeta` / `doubaoAnalysis` 三个方法。

## 5. 链接解析规则

豆包视频分享 URL 包含两个关键 query 参数：

```
share_id=44993598315531266           // 分享标识（仅元数据接口用得上）
video_id=v0269cg10004d82qodiljht41c5h83qg  // 真正的视频 ID（即 vid）
```

正则提取：

```js
var vid     = url.match(/video_id=([^&\s]+)/)[1];
var shareId = (url.match(/share_id=([^&\s]+)/) || ['', ''])[1];
```

## 6. 注意事项

### URL 时效性
返回的 `main_url` 是带签名的 CDN 临时链接，参数里 `dy_q` 是时间戳，
`l` 是 logid，`x-signature`（仅 poster）短期有效。**不要把这个 URL 持久化保存**，
每次需要播放/下载时重新调用接口。

### 反爬
- 必须带 `?aid=497858` query，否则服务端直接吞请求
- UA / Origin / Referer 都不是必需，但建议带上避免被新加的风控规则拦截
- 别在短时间内频繁打同一个 vid，可能触发 IP 维度限流

### 接口字段差异
不要把 `media/get_play_info` 和 `creativity/share/get_video_share_info` 的字段混用：

| 概念 | media 接口 | share 接口 |
|---|---|---|
| 视频 ID 字段名 | `key` | `vid` |
| 视频 URL | `data.media_info[0].main_url` | `data.play_info.main` |
| 原始 URL | `data.original_media_info.main_url` | ❌ 不返回 |
| 封面 | `data.poster_url` | `data.cover_url` |
| prompt | ❌ 不返回 | `data.prompt` |
| 作者 | ❌ 不返回 | `data.user_info.nickname` |

### 当 media/get_play_info 返回 720p 但没有 1080p
有些视频在 `original_media_info` 缺失或 `main_url` 为空，这时直接用
`media_info[0].main_url`（720p 无水印）即可。代码里建议做一次回退：

```js
var downurl = original.main_url || preview.main_url || '';
```

## 7. 失败兜底

如果未来豆包又收口了 `media/get_play_info`，可以排查的方向：

1. 重新跑一遍**反编译流程**：
   - 拉 `https://www.doubao.com/video-sharing?...` 页面 HTML
   - 在 HTML 里找 `lf-flow-web-cdn.doubao.com/.../static/js/` 下的所有 chunk
   - 在 chunk 里搜 `/samantha/` 和 `/creativity/`，找新的接口签名
   - 注意接口对应的 webpack 类方法名（如 `GetMediaPlayInfo`、`GetVideoShareInfo`）
2. 检查接口的**强制 query**，方法是用最小请求集 + 单独添加 query 二分定位
3. 抓取 App 端流量（mitmproxy/Reqable），App 用的接口往往比 Web 多一套，
   且参数集合不同；如果 Web 接口失效 App 接口仍然可用，直接改 UA 模拟 App 即可
