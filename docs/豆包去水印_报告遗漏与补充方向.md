## 一、对用户直觉的回应

你的直觉有合理的一面：

1. **字节内部一定有无水印原片**。AI 生成的是原始帧，编码前才叠加水印；创作者也需要无水印版本用于二次分发。报告 6.5 节也明确论证了这一点。
2. **第三方 APP（便捷下载）确实能返回无水印视频**。这说明存在至少一条服务端路径可以拿到干净文件。

但“有原片”不等于“我们能从公开接口拿到”。目前所有**公开 HTTP API、H5/网页、微信小程序、开源项目**都返回同一份带水印文件，这是有 etag/MD5 证据的。问题不是成本，而是**权限/接口入口**没有暴露给公开客户端。

> 关于成本：在视频编码时叠加水印几乎是零成本（FFmpeg 滤镜一步），真正成本高的是**存储两个版本**。所以字节的策略是只存带水印版本，把无水印原片锁在内部系统或付费/授权通道里。

**未做**：

**风险**：这个服务可能是盗版代理、付费接口，调用它可能涉及版权/ToS 问题。如果要在开源项目里集成，必须谨慎。

## 五、合规与风险提示

1. **公开 API 测试**：对公开端点做少量请求属于技术研究，但请勿高频刷接口。
2. **第三方工具调用**：`便捷下载` 这类服务可能未获字节授权，商用/分发风险高。
3. **APP 反编译**：反编译自家 APP 用于学习一般没问题，反编译竞争对手 APP 可能涉及 ToS/法律风险。
4. **企业/创作者 API**：最合规的路径，值得优先投入。

---

## 六、结论

报告确实没有“穷尽”，但主要不是技术漏洞，而是**入口没暴露**。最有希望的下一步是：

1. **低投入高价值**：补全 `watermark_*` 端点搜索 + 旧版 version_code 回退测试。
2. **中投入**：桌面端/APP 原生抓包，看是否有更高权限接口。
3. **高风险高回报**：调研第三方便捷下载后端的实现方式（但需谨慎合规）。
4. **最合规**：联系豆包创作者/企业支持，申请官方导出能力。
