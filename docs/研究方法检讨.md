# 豆包去水印研究方法检讨

> 2026-07-06 · 刘同学

## 一、我们犯了什么错

### 错误1：API 响应没看全，就下了结论

之前调用 `get_play_info` 时，我们`console.log` 拿到的响应大概长这样：

```json
{
  "data": {
    "play_infos": [
      { "main": "https://...lr=video_gen_watermark_dyn..." }
    ],
    "original_media_info": {
      "main_url": "https://...无水印地址..."   // ← 我们完全没注意到
    }
  }
}
```

我们的视线被 `play_infos[0].main` 吸引了。第一反应是"这个 URL 带水印，我来改参数"。

**问题：一个 JSON 对象有 10 个字段，我们只看了 1 个就想当然认为其他的没用。**

### 错误2：方法论上，我们混淆了"替换参数"和"找新字段"

当我们发现 `lr=video_gen_no_watermark` 返回 403 时，我们的反应是：
- "CDN 不认这个参数" → 换个域名试试？
- "lr 后面可能还有其他值" → 穷举？

**我们从来没有想过：也许 API 里本来就有一个字段存了无水印地址，我们根本不需要改 URL。**

这是思维定式。我们总觉得"客户端只能被动修改参数"，忘了 API 设计者可能已经留了后门。

### 错误3：对 Chrome Debugger API 有偏见

我们之前也想过拦截豆包请求，但又觉得：
- `chrome.debugger` 权限太重，用户看到调试栏会不安
- Manifest V3 的 service worker 用 `fetch` 事件不好拦截
- 注入 MAIN world 的 content script 应该够用了

**这个插件告诉我们：不够。**

它用了 `chrome.debugger.attach` + `Fetch.enable`，直接把所有豆包的网络请求暂停，然后在浏览器内核层面修改响应。我们不是没这个能力，是**主动放弃了这个能力**。

### 错误4：没有建立"失败升级"机制

一个方案失败 → 换新方案，这是我们的一贯流程。

但正确的做法应该是：
- 方案失败 → **回到数据源头，重新审视输入输出**
- 而不是 → 换一个完全不同的方向

我们跳过了"重新审视"这一步，直接跳到"换方向"。

## 二、为什么这些错误会发生

### 1. 我们太想"快速验证"

每次拿到一个 URL，第一反应是"下载试试"，而不是"先完整分析这个 JSON 结构"。

**快速验证的代价是：我们验证了 100 个错误路径，却错过了 1 个正确路径。**

### 2. 我们对"已知信息"过度依赖

之前的研究记录里写着：
> "核心 API：`/samantha/media/get_play_info`（豆包无水印视频）"

这句话本身就是错的。`get_play_info` 不是"无水印视频"的 API，它是"播放信息"的 API。无水印只是它响应里的一个字段。

**我们给 API 贴了标签，然后只关注标签对应的部分。**

### 3. 我们缺乏"数据完整性"意识

正统的研究方法应该是：

```
拿到响应 → 完整打印 → 遍历所有字段 → 逐个测试 → 记录结果
```

我们的做法是：

```
拿到响应 → 找熟悉的字段 → 测试 → 失败 → 换方案
```

**中间少了"完整打印"和"遍历所有字段"两步。**

## 三、正确的做法应该是什么

### 死规矩1：任何 API 调用，先 dump 完整响应

```js
// 错误做法
const url = response.data.play_infos[0].main;
download(url);

// 正确做法
console.log("完整响应:", JSON.stringify(response, null, 2));
// 然后人工检查每个字段
```

### 死规矩2：失败两次以上，必须回到数据源头

不是"换方案"，是"重新看数据"。

### 死规矩3：不要给 API 贴标签

`get_play_info` 不是"无水印 API"，它是"播放信息 API"。它的响应里可能有 20 个字段，我们只看了 1 个。

### 死规矩4：权限不是拒绝的理由

`chrome.debugger` 权限重，但它是正统的浏览器能力。如果它能解决问题，就用它。用户的体验问题可以想办法优化（比如只在豆包页面启用），但不能因为"可能有问题"就放弃整个技术路线。

## 四、这个插件给我们的启示

### 1. 供应链思维

这个插件最聪明的地方不是"改参数"，而是**改配置源头**。

- 豆包前端请求 `samantha/skill/pack` → 拿到视频配置（5秒/10秒）
- 插件拦截这个请求 → 返回本地 JSON（5秒/10秒/15秒）
- 前端拿到假配置 → 展示 15 秒选项
- 用户选择 15 秒 → 前端把 15 秒传给后端
- 后端生成 15 秒视频 → 返回无水印地址

**它没有"破解"任何东西，它只是让前端以为"15 秒是官方支持的时长"。**

### 2. 多层防御

这个插件用了 7 层注入来确保 15 秒配置生效：
1. Chrome Debugger 拦截响应
2. 本地 skill-pack 文件替换
3. 动态 JSON 补丁
4. Webpack store hook
5. JSON.parse 劫持
6. 请求体修改
7. UI 层替换

任何一层失效，还有其他层兜底。我们之前总是"一层不行就换方案"，没有"多层保险"的概念。

### 3. 对 API 字段的敬畏

`original_media_info.main_url` 这个字段名说明什么？
- `original` = 原始的
- `media_info` = 媒体信息
- `main_url` = 主地址

**API 设计者明确告诉你：这是"原始媒体"的地址。** 我们没看，是我们的问题。

## 五、总结

| 我们的做法 | 应该做的 |
|-----------|---------|
| 拿到 URL 就下载测试 | 先 dump 完整响应，遍历所有字段 |
| 失败就换方案 | 失败两次以上，回到数据源头重新审视 |
| 给 API 贴标签 | 把 API 当黑盒，不预设它"应该"返回什么 |
| 回避重权限 API | 用能力解决问题，再优化体验 |
| 单层验证 | 多层保险，任何一层失效有其他层兜底 |

**一句话：不是思路不对，是我们连 API 完整返回都没看全。**
