# 豆包视频无水印链接探索计划

> 生成时间：2026-07-06  
> 目标：在现有 `hope0719/doubao-video-watermark-analysis` 项目基础上，梳理已验证结论、明确剩余可行方向，并给出可执行的行动计划。

---

## 一、项目现状一句话总结

**截至 2026-07-05，所有公开 HTTP API、开源项目、参数操控、域名变换、登录态调用，都只能拿到“带水印”的豆包视频文件。** 项目目前处于“已穷尽已知客户端路径”的状态。

---

## 二、已验证的核心结论

### 2.1 技术根因：水印在服务端编码阶段已嵌入

```
AI 生成（无水印原片）
    ↓
H.264/H.265 编码器 像素级叠加水印
    ↓
TOS 对象存储 只存一份带水印文件
    ↓
CDN 分发所有 URL（参数只影响缓存签名/统计）
    ↓
客户端拿到的是带水印文件
```

### 2.2 已穷尽的公开路径

| 方向 | 关键结论 | 是否排除 |
|------|---------|:--------:|
| `POST /samantha/media/get_play_info` 取 `original_media_info.main_url` | 现在与 `media_info[0].main_url` 完全一致，且带 `lr=video_gen_watermark_dyn` | ✅ |
| `POST /creativity/share/get_video_share_info` | 返回 `v9/v26` 不同编码版本，但同样带水印 | ✅ |
| 登录态 + `credentials: include` | 与未登录态返回同一文件 | ✅ |
| 修改 `lr` 参数（none/no_watermark/origin 等） | CDN 缓存 key 不含查询参数，etag 不变 | ✅ |
| 去掉 `ft` / `cs` / `download` 等参数 | 同样 etag 不变，或 403 | ✅ |
| 切换 CDN 域名 `v9`/`v26`/`show.douyinvod` | 仅对应不同编码配置，均有水印 | ✅ |
| 15+ 开源项目源码验证 | 全部使用相同 API，返回相同带水印文件 | ✅ |
| 微信小程序端抓包 | 抓到的视频请求同样带水印 | ✅ |
| 第三方“去水印小程序” | 要么是 API 代理（仍有水印），要么是 AI 修复/裁剪 | ✅ |

### 2.3 为什么图片能去、视频不能去

| 维度 | 图片 | 视频 |
|------|------|------|
| 水印位置 | 前端叠加层（CSS/Canvas） | 服务端编码像素烘焙 |
| 原始文件 | 服务器同时返回无水印原图 `image_ori_raw.url` | 服务器只存带水印版本 |
| 绕过方式 | 提取原始 URL 即可 | 无 URL/API 可绕过 |

## 三、剩余可行方向（按可行性排序）

### 方向 A：画面裁剪（最短期可落地）

- **原理**：豆包水印一般固定在画面边缘，通过裁剪可去除。
- **实现**：FFmpeg / Canvas / WebCodecs API 在浏览器端或本地处理。
- **优点**：不依赖服务端，实现简单。
- **缺点**：损失边缘像素，需预先知道水印位置。
- **建议动作**：
  1. 用现有工具下载几个不同视频样本。
  2. 用 FFmpeg 测试裁剪参数（如 `crop=1240:720:40:0` 等）。
  3. 在 Edge 插件中增加“一键裁剪下载”按钮。

### 方向 B：AI 后处理去水印（效果最好，成本最高）

- **原理**：逐帧检测水印区域并用 AI inpainting 修复。
- **可选工具**：Morph Studio、HitPaw、UnscreenVideo、ComfyUI inpainting 节点。
- **优点**：不依赖 API，可处理动态水印、中央水印。
- **缺点**：需要 GPU / 付费额度，处理慢，重建区域非原始像素。
- **建议动作**：
  1. 评估 Morph Studio 免费额度是否足够日常场景。
  2. 如需求量大，可调研自托管模型（如 ProPainter、Track-Anything）做批量处理。

### 方向 C：调用第三方“去水印服务”后端（如 便捷下载 / 类似工具）

- **背景**：项目 `app-decompilation-report.md` 指出，某 Android 下载工具在服务端完成了无水印解析，说明存在服务端能力。
- **风险**：
  - 需要注册、付费、可能涉及版权/ToS 问题。
  - 服务端逻辑不可见，无法保证稳定性。
  - 依赖第三方服务，有隐私泄露风险（视频上传）。
- **建议动作**：
  1. 仅作为调研方向，收集公开文档/价格/用户反馈。
  2. 不推荐在开源项目中直接集成，避免法律与合规风险。

### 方向 D：继续挖掘豆包内部 API（低概率，但可监控）

- **线索**：JS bundle 中仍存在 `watermark_download`、`watermark_task` 等端点定义，但缺少任务提交入口。
- **建议动作**：
  1. 每周/每月运行一次监控脚本，检查 `original_media_info.main_url` 是否与 `media_info[0].main_url` 再次产生差异。
  2. 监控 `lr` 参数是否从 `video_gen_watermark_dyn` 变成其他值。
  3. 关注豆包版本更新后 JS bundle 是否新增端点。

### 方向 E：官方会员/导出渠道（推荐用于商用）

- **原理**：豆包面向创作者或会员可能提供无水印导出能力。
- **优点**：无损、合法、稳定。
- **缺点**：可能付费。
- **建议动作**：
  1. 在豆包 APP/网页中查看“导出/下载”高级选项。
  2. 联系豆包客服或查看创作者中心是否有企业级 API。

## 四、推荐行动方案

### 方案 1：先让现有工具“可用”——保留图片去水印，视频改为裁剪+AI修复

- Edge 插件：保留图片提取 `image_ori_raw.url` 的功能（仍有效）。
- 视频：增加下载按钮，下载后提供“裁剪去水印”选项。
- 后端/脚本：集成 FFmpeg 裁剪或调用 AI 去水印服务。

### 方案 2：建立监控，等待官方 API 变化

- 自动化脚本每天调用一次 `get_play_info`，比较 `original_media_info` 与 `media_info` 的 URL 是否重新分化。
- 一旦分化，立即触发通知，可快速恢复无水印下载能力。

### 方案 3：调研第三方服务端方案（低风险调研）

- 收集公开资料，评估是否可接入合规的第三方去水印服务。
- 不直接逆向、不破解、不大量调用，避免合规风险。

## 五、可立即执行的脚本/工具

### 5.1 下载带水印视频（现有工具可直接使用）

```bash
cd doubao-video-watermark-analysis/tools
pip install -r requirements.txt
python doubao_video_downloader.py "https://www.doubao.com/video-sharing?video_id=xxx"
```

### 5.2 验证当前 API 是否恢复无水印（监控用）

```bash
python ../proof/test_api.py
```

### 5.3 快速裁剪去水印（FFmpeg）

```bash
# 假设水印在右侧，裁剪掉 40 像素
ffmpeg -i input.mp4 -vf "crop=1240:720:0:0" -c:a copy output.mp4
```

## 六、注意事项与合规边界

1. **本项目的所有研究应遵守豆包平台服务条款与相关版权法规。**
2. **图片去水印**在技术上可绕过前端叠加层，风险相对可控；**视频无水印原片**涉及服务端资源，不建议通过非官方渠道大规模获取。
3. 对于第三方“去水印 APP”的服务端能力，建议仅做调研，不直接集成到开源项目中，避免连带责任。
4. 推荐优先使用**官方导出/会员**、**画面裁剪**、**AI 修复**等合法替代方案。

---

## 七、下一步建议

1. **如果目标是“尽快能用”**：实现视频下载 + 边缘裁剪，优先交付一个可裁剪下载的 Edge 插件版本。
2. **如果目标是“真正无损”**：调研豆包官方会员/创作者导出渠道，或接入 Morph Studio 等 AI 修复工具。
3. **如果目标是“持续跟踪”**：部署 API 监控脚本，等待豆包服务端恢复 `original_media_info` 无水印字段。
